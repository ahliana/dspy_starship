The Cell-by-Cell Guide

Every live cell in dspy_starship_01.ipynb, explained slowly. Written for someone new to DSPy and still warming up to Python. For each cell: the exact code, what you will see when it runs, what each line is doing, why it is built that way, and the questions it tends to raise.

During Q&A: click the sidebar to jump, or Ctrl+F any term. Every DSPy word is in the glossary at the bottom.

Before any code

The big picture

The problem. Most teams improve a language model by editing the prompt by hand: add a rule, rerun, eyeball the output, add another rule. That is the crop duster. Low altitude, exhausting, and when a new model ships, your carefully tuned wording often stops working.

The DSPy idea. Stop writing prompts. Instead, write a small program in three moves:

  1. Declare a signature: the inputs and outputs of your task. Nothing else.
  2. Compose with a module: Predict, ChainOfThought, or ReAct, which decide how the model is asked.
  3. Optimize: hand DSPy a few labeled examples and a metric, and let an optimizer write a better prompt than you would by hand.

The x-ray glasses. dspy.inspect_history(n=1) prints the exact prompt DSPy sent to the model on the last call. You never write these prompts, but you can always see them. The notebook puts the glasses on six times; this guide flags each moment.

CELL 0 Pick your model

Reads your API key from a file and chooses which models the whole notebook will use. The only decision you make all night.

provider selector
import os
from dotenv import load_dotenv

load_dotenv()

if os.environ.get("ANTHROPIC_API_KEY"):
    PROVIDER = "anthropic"
    FAST_MODEL = "anthropic/claude-haiku-4-5-20251001"
    BIG_MODEL = "anthropic/claude-sonnet-4-6"
    FAST_LABEL, BIG_LABEL = "Claude Haiku 4.5", "Claude Sonnet 4.6"
elif os.environ.get("OPENAI_API_KEY"):
    PROVIDER = "openai"
    FAST_MODEL = "openai/gpt-4o-mini"
    BIG_MODEL = "openai/gpt-4o"
    FAST_LABEL, BIG_LABEL = "GPT-4o mini", "GPT-4o"
else:
    raise RuntimeError(
        "No API key found. Add ANTHROPIC_API_KEY or OPENAI_API_KEY to your .env file."
    )

print(f"Provider: {PROVIDER}")
print(f"Fast model: {FAST_LABEL}  ({FAST_MODEL})")
print(f"Bigger model, used once in the model-swap demo: {BIG_LABEL}  ({BIG_MODEL})")
What you see when it runs
Three lines naming your provider, your fast everyday model, and the bigger model that appears once in cell 9. On an Anthropic key: Provider: anthropic and Claude Haiku 4.5 as the fast model.
LineWhat it is doing, in plain English
import osBrings in Python's standard toolbox for talking to the operating system. We use it to read environment variables.
from dotenv import load_dotenvImports one function from the python-dotenv package. "From this box, take out just this one tool."
load_dotenv()Opens the .env file sitting next to the notebook and loads each KEY=value line into the environment, as if you had typed them into the terminal. Your key lives in that file, never in the code.
os.environ.get("ANTHROPIC_API_KEY")Looks up that variable. Returns the value if present, or None if absent. In an if, a real value counts as true and None counts as false, so this reads "if the Anthropic key exists."
if / elif / elseA three-way fork: Anthropic key wins if present, otherwise OpenAI, otherwise stop.
FAST_MODEL = "anthropic/claude-..."Model choice is literally a string: provider name, slash, model name. This string is the ONLY place the provider exists in the entire notebook.
raise RuntimeError(...)Stops the notebook immediately with a readable message instead of failing mysteriously three cells later.
print(f"Provider: {PROVIDER}")An f-string: the f before the quote lets you drop variables into the text with {curly braces}.
Why it is built this way
Two reasons. Security: the key stays in .env, which is gitignored, so it can never be committed or shown on screen. Portability: because the model is just a string, the audience can run the identical notebook on an OpenAI key, and nothing else changes. DSPy reaches any provider through an adapter library called LiteLLM; to use a local model via Ollama you would write "ollama/llama3".
If someone asks
  • "What if I have both keys?" Anthropic wins, because the if checks it first.
  • "Is my key sent anywhere?" Only to the provider you are calling, over HTTPS, the same as any API client.

CELL 1 Connect DSPy to your model

Builds the model handle, makes it the default for everything below, and proves the connection with one print.

setup
import dspy

lm = dspy.LM(FAST_MODEL, max_tokens=800, temperature=0.0)
dspy.configure(lm=lm)
print(f"Starship online. Flying {FAST_LABEL}.")
What you see when it runs
Starship online. Flying Claude Haiku 4.5. If instead you get an authentication error, the key in .env is wrong or still has placeholder text in it.
PieceWhat it is doing
dspy.LM(FAST_MODEL, ...)Creates a language-model handle: an object that knows which model to call and with what settings. Nothing is sent to the model yet.
max_tokens=800Caps how long each response may be. Tokens are the chunks models read and write, roughly three quarters of a word each. The cap keeps responses fast and spending tiny.
temperature=0.0Turns off sampling randomness. Same input, same output, every run. Deterministic behavior is what makes the accuracy numbers later honest and repeatable on stage.
dspy.configure(lm=lm)Registers this handle as the global default. Every DSPy call below uses it automatically, so you never pass the model around by hand.
Why it is built this way
Configure once, use everywhere. When cell 9 wants a different model for two calls, it will temporarily override this default inside a with block instead of reconfiguring, which is the polite way to A/B models.
If someone asks
  • "Does temperature 0 make the model correct?" No. It makes it consistent. It will give the same wrong answer every time too, which is exactly what you want while measuring.
  • "What happens at the token cap?" The response is cut off at 800 tokens. For one-word classifications this cap is never reached.

CELL 2 Your first signature

The whole program is one line. Nobody wrote a prompt.

string signature
classify = dspy.Predict("text -> sentiment: str")
result = classify(text="The launch was successful and the crew is in great spirits.")
print(result.sentiment)
What you see when it runs
Something like positive. The model read the sentence, DSPy asked it for a sentiment field, and you printed that field.
PieceWhat it is doing
"text -> sentiment: str"A signature: the task's contract. Left of the arrow, the input field name. Right of the arrow, the output field name and its type. Read it as "given text, produce a sentiment, as a string."
dspy.Predict(...)Wraps the signature in the simplest module, which just asks the model once, directly. The result is a callable program stored in classify.
classify(text=...)Runs the program. You pass inputs by name (a keyword argument), matching the field name in the signature.
result.sentimentThe response comes back as a Prediction object with one attribute per output field. DSPy already parsed the model's reply into fields; you never scrape text yourself.
Why it is built this way
This is the "declare" move. You state WHAT the task is; DSPy handles HOW to ask. Behind the scenes it generated a real prompt from those five visible characters of task description. Which raises the obvious question: what did it actually send? That is cell 3.

CELL 3 X-ray glasses, moment 1

See the prompt you did not write.

the reveal
dspy.inspect_history(n=1)
X-ray moment 1 of 6
inspect_history(n=1) prints the last exchange DSPy had with the model: the full prompt out, the raw response back. Reading it top to bottom you will find three things you never typed:
  • A system message stating the task and defining each input and output field.
  • A user message carrying your actual input text.
  • Field markers like [[ ## sentiment ## ]], which tell the model exactly where to put each answer, and which DSPy then uses to parse the reply back into result.sentiment.
Why it matters
This one screen is the paradigm shift: the prompt is generated output, not source code. You edit signatures; the library edits prompts. It also makes DSPy a glass box rather than a black box. Whenever an output comes back empty or weird, put the glasses on first; the problem is usually visible in ten seconds.
If someone asks
  • "Can I edit that generated prompt directly?" You do not. You influence it three ways: field names, field descriptions, and the signature docstring. Later, the optimizer rewrites it for you, guided by data.
  • "What does n=1 mean?" Show the last 1 call. n=2 shows the last two, which cell 7 uses to compare prompts side by side.

CELL 4 Class-form signatures

The grown-up version of a signature: typed fields, descriptions, and a docstring that becomes instruction.

typed signature
class TransmissionReport(dspy.Signature):
    """Classify a transmission from a spacecraft."""
    transmission: str = dspy.InputField()
    severity: str = dspy.OutputField(desc="One of: nominal, advisory, urgent")
    summary: str = dspy.OutputField(desc="One sentence summary")

report = dspy.Predict(TransmissionReport)
out = report(transmission="Houston, we have a problem. Main bus B undervolt.")
print(f"Severity: {out.severity}")
print(f"Summary: {out.summary}")
What you see when it runs
Two labeled lines, something like Severity: urgent and a one-sentence summary of the power problem. Two structured outputs from one call.
PieceWhat it is doing
class TransmissionReport(dspy.Signature):A Python class is a blueprint. This one inherits from dspy.Signature, which is how DSPy recognizes it as a task description rather than ordinary code.
"""Classify a transmission..."""The docstring. In normal Python it is documentation; in a DSPy signature it is sent to the model as the task instruction. Write it like an instruction, because it is one.
transmission: str = dspy.InputField()One input field. The : str part is a type hint, a label saying what kind of value this is.
severity: str = dspy.OutputField(desc=...)An output field with a description. That desc text flows straight into the generated prompt, so "One of: nominal, advisory, urgent" acts as a soft constraint on the answer.
Why it is built this way
The string form from cell 2 is for quick experiments. Real programs want several typed fields, guidance per field, and an instruction, and the class form carries all of that while behaving identically. Adding a whole new structured output later is a one-line edit to the class, not a prompt rewrite.
If someone asks
  • "Is desc a guarantee the model complies?" No, it is strong guidance. If you need a hard guarantee, check the value in code or make the metric punish violations; the optimizer then learns to respect it.

CELL 7 Predict vs ChainOfThought

Same signature, two modules, visibly different behavior. Swap one word, gain a reasoning step.

module comparison
class ExplainCode(dspy.Signature):
    """Explain what this Python code does in plain English."""
    code: str = dspy.InputField()
    explanation: str = dspy.OutputField()

snippet = "def fib(n):\n    return n if n <= 1 else fib(n-1) + fib(n-2)"

print("Predict says:")
print(dspy.Predict(ExplainCode)(code=snippet).explanation)
print("\nChainOfThought says:")
result = dspy.ChainOfThought(ExplainCode)(code=snippet)
print(f"Reasoning: {result.reasoning}\n")
print(f"Explanation: {result.explanation}")

# X-RAY GLASSES MOMENT 2: peek at both prompts
print("\n--- Both prompts DSPy sent ---")
dspy.inspect_history(n=2)
What you see when it runs
Two explanations of the same Fibonacci function. The ChainOfThought one is preceded by a Reasoning: paragraph that you never declared: the module added that output field by itself. Then both raw prompts print.
PieceWhat it is doing
snippet = "def fib(n):\n ..."A tiny recursive Fibonacci function stored as a string. The \n is a newline character inside the string, so the model sees properly formatted code.
dspy.Predict(ExplainCode)(code=snippet)Build the program and call it in one breath, then read .explanation off the result. Chaining like this is common Python shorthand.
dspy.ChainOfThought(ExplainCode)The SAME signature wrapped in a different module. ChainOfThought instructs the model to think step by step first, and automatically adds a reasoning output field to carry that thinking.
dspy.inspect_history(n=2)Glasses on, last TWO calls this time, so you can diff what the second module changed.
X-ray moment 2 of 6
Compare the two printed prompts. The ChainOfThought one contains an instruction you never wrote, roughly "think step by step," plus the extra reasoning field placed before the answer. Modules are reusable asking-strategies, applied by changing one word.
If someone asks
  • "When should I prefer ChainOfThought?" Multi-step or tricky reasoning tasks. For one-word classification it mostly adds tokens and time, which is why the optimizer section uses plain Predict.
  • "What is ReAct?" The third module named in the talk: it lets the model call tools you provide (search, calculator, your functions) in a loop until it can answer. Same signature idea, more capable engine.

CELL 8 A real tool: the code reviewer

Signature plus module equals working software: score, issues, and a rewrite in one call.

composed tool
class CodeReviewSig(dspy.Signature):
    """Review Python code and return structured findings."""
    code: str = dspy.InputField()
    quality: int = dspy.OutputField(desc="Quality score from 1 to 10")
    issues: str = dspy.OutputField(desc="Newline-separated list of specific issues")
    improved: str = dspy.OutputField(desc="The code rewritten with the issues fixed")

reviewer = dspy.ChainOfThought(CodeReviewSig)

code_to_review = """
def calculate_total(items):
    total = 0
    for item in items:
        total = total + item.price
    return total
"""

result = reviewer(code=code_to_review)
print(f"Quality: {result.quality}/10")
print(f"Issues:\n{result.issues}")
print(f"\nImproved:\n{result.improved}")
What you see when it runs
A quality score around 6/10, a list of concrete issues (no docstring, could use sum(), no input validation), and a rewritten function fixing them. A real code review, structured, from one call.
PieceWhat it is doing
quality: int = dspy.OutputField(...)An output typed as an integer. DSPy parses the model's reply into an actual int, so result.quality is a number you could compare or sort by, not a string.
reviewer = dspy.ChainOfThought(CodeReviewSig)Reviewing code benefits from thinking first, so this tool picks the reasoning module. That choice is the "compose" decision.
code_to_review = """ ... """Triple quotes make a multi-line string, the natural way to hold a code block inside code.
Why it is built this way
Notice what is absent: no prompt writing, and no output parsing. Three structured fields come back ready to use. That absence is the productivity story: ten lines is the entire tool, and it is model-portable, which cell 9 proves immediately.

CELL 9 Swap models in one line

The same reviewer runs on a bigger model with zero changes to the program.

the portability payoff
big = dspy.LM(BIG_MODEL, max_tokens=1000)
with dspy.context(lm=big):
    big_result = reviewer(code=code_to_review)

print(f"{FAST_LABEL} said:")
print(result.improved)
print("\n---\n")
print(f"{BIG_LABEL} said:")
print(big_result.improved)
What you see when it runs
Both rewrites side by side. The bigger model's version is usually more thorough: docstring, type hints, sometimes error handling. Same program, different engine, visible quality difference.
PieceWhat it is doing
big = dspy.LM(BIG_MODEL, ...)A second model handle, pointing at the bigger model chosen back in cell 0.
with dspy.context(lm=big):A with block creates a temporary zone. Inside it, DSPy's default model is big; the moment the block ends, the original default from cell 1 is restored automatically. Nothing to undo by hand.
big_result = reviewer(...)The very same reviewer object from cell 8. It was never told about models; it just runs on whatever the current default is.
Why this is the metaphor's proof
In hand-tuned prompting, a model upgrade means re-testing and often rewriting the prompt. Here the model is a parameter. When something better or cheaper ships, you change one string, and if you want the last few points of accuracy back, you re-run the optimizer. The program does not change. That is the crop duster versus the starship in one cell.

CELL 10 The task, the data, and the baseline

Classify spacecraft transmissions by urgency, where the right answer follows a team convention no model can guess.

Read this before the code. The task sorts transmissions into routine, attention, or urgent. The trap is a workplace convention: a confirmed threat is urgent, but a merely suspected or unconfirmed hazard is only attention, because this team's first move is to investigate, not to scramble. Common sense says "possible fire, escalate!" The convention says "possible means verify first." A convention lives in labeled data, not in world knowledge, and that is precisely where an optimizer earns its keep.

signature, data, metric, baseline (abridged: 4 of 12 train and 3 of 10 test examples shown; the notebook has the full lists)
class ClassifyTransmission(dspy.Signature):
    """Classify a spacecraft transmission by urgency."""
    transmission: str = dspy.InputField()
    urgency: str = dspy.OutputField(desc="One of: routine, attention, urgent")

# Training set: 12 examples that ENCODE the convention.
train = [
    dspy.Example(transmission="Suspected coolant leak, still confirming whether it is real.",
                 urgency="attention").with_inputs("transmission"),
    dspy.Example(transmission="Confirmed cabin depressurization, crew on emergency oxygen.",
                 urgency="urgent").with_inputs("transmission"),
    dspy.Example(transmission="Scheduled fire drill completed, all crew accounted for.",
                 urgency="routine").with_inputs("transmission"),
    dspy.Example(transmission="Unidentified vibration of unknown origin, investigating.",
                 urgency="attention").with_inputs("transmission"),
    # ... 8 more in the notebook: 3 urgent, 2 routine, 3 attention ...
]

# Held-out test set: 10 examples the optimizer NEVER sees.
# The traps are unconfirmed hazards whose true label is attention.
test = [
    dspy.Example(transmission="Smoke detector triggered; verifying whether it is a real fire or a faulty unit.",
                 urgency="attention").with_inputs("transmission"),
    dspy.Example(transmission="Confirmed reactor coolant leak, sealing the aft compartment now.",
                 urgency="urgent").with_inputs("transmission"),
    dspy.Example(transmission="Morning coffee service commencing on the flight deck.",
                 urgency="routine").with_inputs("transmission"),
    # ... 7 more in the notebook ...
]

def match(example, pred, trace=None):
    return example.urgency.strip().lower() == pred.urgency.strip().lower()

def measure_accuracy(program, examples, label):
    correct = 0
    misses = []
    for ex in examples:
        pred = program(transmission=ex.transmission)
        if match(ex, pred):
            correct += 1
        else:
            misses.append((ex.urgency, pred.urgency.strip().lower(), ex.transmission))
    accuracy = correct / len(examples)
    print(f"{label}: {correct}/{len(examples)} correct ({accuracy*100:.0f}%)")
    for gold, got, text in misses:
        print(f"  MISS: expected {gold}, got {got}  |  {text[:58]}")
    if not misses:
        print("  No misses.")
    return accuracy

# Baseline: no optimization, no examples, just the signature
baseline = dspy.Predict(ClassifyTransmission)
print("Baseline test case:", baseline(transmission="Cabin fire, crew evacuating.").urgency)
print("\n--- X-RAY GLASSES MOMENT 3: baseline prompt ---")
dspy.inspect_history(n=1)

print("\nNow score the bare baseline on all 10 test transmissions:")
baseline_accuracy = measure_accuracy(baseline, test, "Plain Predict baseline")
What you see when it runs
The single test case comes back urgent (a confirmed fire is easy). Then the bare prompt prints, then the score: Plain Predict baseline: 6/10 correct (60%), with each miss printed on its own line. The misses are exactly the convention cases: unconfirmed hazards the model escalates to urgent when the team wants attention.
PieceWhat it is doing
dspy.Example(transmission=..., urgency=...)One labeled example: an input paired with the correct answer. This is training data in its smallest form.
.with_inputs("transmission")Tells DSPy which field is the input. Everything not named there (here, urgency) is treated as the label to learn from.
train vs testTwo separate lists with zero overlap. The optimizer will only ever touch train. test stays locked away so the final scores are honest; measuring on data the optimizer saw would be grading with the answer key open.
def match(example, pred, trace=None):The metric: the definition of success as a function. It lowercases and trims both answers and checks equality. The extra trace=None parameter is there because DSPy's optimizers call metrics with a third argument during compilation.
measure_accuracy(...)Plain Python, no DSPy magic: loop over examples, run the program, count matches, collect misses, print a percentage. Printing each miss is deliberate showmanship: the audience sees WHICH cases fail, not just a number.
accuracy = correct / len(examples)Correct answers divided by total, so 6 of 10 is 0.6. The f-string later formats it as a percent with {accuracy*100:.0f}, meaning "times 100, no decimal places."
baseline = dspy.Predict(ClassifyTransmission)The "before" picture: the signature alone, no examples, no tuning. Every improvement gets measured against this.
X-ray moment 3 of 6
The baseline prompt is nearly bare: field definitions and formatting markers, no guidance about what urgent means. Sixty percent is what raw model common sense buys on this task. Remember the number; two contenders are about to try to beat it.
If someone asks
  • "Why is the baseline as high as 60?" Confirmed catastrophes and coffee service are easy calls for common sense. Only the convention cases fool it.
  • "Twelve training examples seems tiny." That is a feature of DSPy's few-shot optimizers: 5 to 20 labeled examples is a normal, useful range. You almost certainly have that many lying around for any real task.

CELL 11 The hand-tuned prompt (the crop duster)

A genuinely good manual prompt, every trick included, with one guessed rule that sinks it.

the challenger
HAND_TUNED_PROMPT = """You are an expert flight controller with 20 years of experience \
classifying spacecraft transmissions. Categorize each transmission by urgency.

CATEGORIES:
- routine: normal operations, scheduled activities, no anomalies
- attention: a minor anomaly that is clearly under control
- urgent: any threat to crew or vehicle

SAFETY POLICY (this is critical, follow it strictly):
1. Any mention of fire, smoke, leak, pressure loss, power loss, or radiation is URGENT.
2. If a hazard is not yet confirmed, treat it as URGENT anyway. Never wait for confirmation.
3. When in doubt, escalate. It is always safer to over-report.

EXAMPLES:
- "Cabin humidity slightly elevated, within acceptable range." -> attention
- "Confirmed hull breach, sealing bulkheads now." -> urgent
- "Daily science experiment completed on schedule." -> routine

Reply with EXACTLY one word: routine, attention, or urgent. No punctuation, no explanation.

Transmission: {transmission}

Urgency:"""

# Wrap the hand-tuned prompt in a callable program with the same interface
class HandTunedProgram:
    def __call__(self, transmission):
        from types import SimpleNamespace
        # Send the hand-tuned prompt directly to the LM
        response = lm(prompt=HAND_TUNED_PROMPT.format(transmission=transmission))
        # LM returns a list of strings, take the first, strip
        raw = response[0].strip().lower()
        # Normalize: sometimes the model adds a period or extra words
        for word in ["routine", "attention", "urgent"]:
            if word in raw:
                return SimpleNamespace(urgency=word)
        return SimpleNamespace(urgency=raw)

hand_tuned = HandTunedProgram()

print("Running the hand-tuned prompt on 10 test transmissions...")
print()
hand_tuned_accuracy = measure_accuracy(hand_tuned, test, "Hand-tuned prompt")
print()
print("--- X-RAY GLASSES MOMENT 4: the last hand-tuned prompt sent ---")
dspy.inspect_history(n=1)
What you see when it runs
Hand-tuned prompt: 6/10 correct (60%). Same score as the bare baseline, but look at the printed misses: they are DIFFERENT mistakes. The baseline missed by ignorance; this prompt misses by policy, escalating every suspected hazard to urgent because rule 2 told it to.
PieceWhat it is doing
HAND_TUNED_PROMPT = """You are an expert..."""Classic prompt engineering, all of it: a role ("20 years of experience"), category definitions, a strict policy, few-shot examples, and a rigid output format demand. This is hours of somebody's honest work, compressed.
the backslash at the end of line 1A line continuation inside the string, purely cosmetic so the source stays readable.
{transmission} ... .format(...)The placeholder gets filled with each actual transmission at call time. This is the older cousin of the f-string.
class HandTunedProgram: def __call__(...)Defining __call__ makes an object usable like a function. The point is fairness: wrapping the prompt this way gives it the same calling interface as a DSPy program, so the exact same measure_accuracy judges both on the exact same test set. No home-field advantage.
response = lm(prompt=...)Calls the model directly with the raw prompt, bypassing signatures entirely. The handle returns a list of response strings; response[0] is the first.
the normalize loopHand-rolled output parsing: hunt for one of the three category words inside the reply. Notice you get this chore for free with DSPy and have to write it yourself without it.
SimpleNamespace(urgency=word)A quick throwaway object with an .urgency attribute, mimicking the shape of a DSPy Prediction so the scorer can read it identically.
X-ray moment 4 of 6
The glasses print the hand-tuned prompt exactly as the model received it. Why the good prompt fails, and why that is the point
Rule 2, "if a hazard is not yet confirmed, treat it as URGENT anyway," is a sensible instinct and exactly wrong for this team, whose convention says unconfirmed means investigate, label it attention. The author guessed a policy the data disagrees with. That is the deep lesson of the whole talk: a prompt encodes what you guessed; labeled examples encode what is actually true. No amount of wordsmithing fixes a wrong guess you do not know you made.
If someone asks
  • "Isn't escalating unconfirmed hazards the RIGHT call in real life?" Often yes! The point is not that the policy is dumb; it is that the spec lives in the data, and the prompt author never saw the spec. Swap in your own domain: your team's ticket-triage rules, your legal team's redlines. Conventions are everywhere and models cannot guess them.
  • "Could I just add the convention to the prompt?" If you knew it, yes. The optimizer's trick is discovering it from examples even when nobody wrote it down.

What you see when it runs
Compilation finishes in a few seconds. Then: DSPy BootstrapFewShot: 9/10 correct (90%), the three-line comparison, and STARSHIP WINS. The glasses then show the prompt now carries worked examples that were not there before.
PieceWhat it is doing
from dspy.teleprompt import BootstrapFewShotOptimizers live in the teleprompt module (the historical name for DSPy's optimizer family).
BootstrapFewShot(metric=match, max_bootstrapped_demos=4)Configures the optimizer: use OUR definition of success (match) to judge, and attach at most four demonstrations to the prompt.
optimizer.compile(student=..., trainset=train)The actual mechanism, and it is beautifully simple: run the student program on each training example, keep the ones it answers CORRECTLY (per the metric), and pack those correct input-answer pairs into the prompt as worked examples, called demonstrations. time.time() ... elapsedStopwatch bookkeeping so the audience sees this took seconds, not hours.

Why it beats the hand-tuned prompt
The demonstrations are drawn from the training set, and the training set encodes the real convention: confirmed leaks labeled urgent, suspected leaks labeled attention, sitting side by side. The model reads those worked examples and applies the pattern. Nobody wrote better rules; labeled examples defined the boundary that the hand-tuned author guessed wrong. Thirty points from twelve labels and a two-line optimizer.

The final wearing of the glasses shows MIPROv2's full prompt: its rewritten instruction plus the 16 demonstrations it selected. That clean printout is the moment the room reads a machine-written prompt that beats the human one.
What MIPROv2 actually does, and why it ran before the show
BootstrapFewShot only selects demonstrations. MIPROv2 searches over two things at once: candidate instructions (it drafts many phrasings, using a bigger model as the proposal writer) and candidate demonstration sets, testing combinations against your metric and keeping the winners; the search is guided so promising combinations get explored first. That thoroughness needs more data and more model calls than a live demo allows, so it was compiled ahead of time, in a few minutes, on a larger 40-example training set with a separate 20-example validation split, then saved. The recipe is scripts/run_miprov2.py in the repo. Crucially, the 10 test examples were never shown to either optimizer.
If someone asks
  • "Why does a written instruction beat demonstrations alone?" Demonstrations teach by pattern, and four examples can only show so many edges. An explicit rule generalizes to cases no demo covered. Best of all, MIPROv2 delivers both at once.
  • "How much did compiling cost?" Under a dollar of API calls, about five minutes. The artifact is committed to the repo, so nobody has to rebuild it to run the notebook.
  • "Is 100% real?" On these ten held-out examples, at temperature 0, yes and reproducibly. Ten examples is a demo-sized test set; the claim is "it learned the convention," not "it is a perfect classifier."

Reference A

Glossary: DSPy and LLM terms

artifact
An optimized program saved to a file (here, miprov2_artifact.json): the instruction plus chosen demonstrations as JSON. Load it to restore the optimized behavior exactly, which is also how you deploy.
baseline
The untuned starting point you measure everything against. Here: plain Predict with no examples, scoring 60%.
BootstrapFewShot
The fast optimizer. Runs your program on the training set, keeps the examples it gets right (per your metric), and packs them into the prompt as demonstrations. Seconds to run.
ChainOfThought
A module that instructs the model to reason step by step before answering, automatically adding a reasoning output field. Costs extra tokens; helps on multi-step problems.
compile
DSPy's word for running an optimizer: optimizer.compile(student=..., trainset=...) returns an improved copy of your program.
demonstration (demo)
A worked example (input plus correct answer) placed inside the prompt for the model to imitate. What BootstrapFewShot selects and what optimized.demos holds.
few-shot / zero-shot
Few-shot means the prompt includes worked examples; zero-shot means it includes none. The baseline is zero-shot; the optimized programs are few-shot.
held-out test set
Labeled examples locked away from the optimizer and used only for final scoring. Keeping them unseen is what makes an accuracy claim honest.
inspect_history
The x-ray glasses. dspy.inspect_history(n=1) prints the last prompt sent and response received. Your first debugging move in DSPy.
instruction
The natural-language task description at the top of a generated prompt. Comes from your signature's docstring, until MIPROv2 rewrites it into something better.
LiteLLM
The adapter library DSPy uses to reach every model provider through one interface. The provider lives in the model string: "anthropic/...", "openai/...", "ollama/...".
LM (dspy.LM)
A handle to a language model plus its settings (which model, max_tokens, temperature). Set a default once with dspy.configure.
metric
A function that scores a prediction against a labeled example, returning True/False or 0 to 1. It is your definition of success, and it steers every optimizer.
MIPROv2
The thorough optimizer. Drafts many candidate instructions (using a bigger model as writer), pairs them with candidate demonstration sets, and searches for the combination that scores best on your metric. Minutes, not seconds; run it offline and save the artifact.
module
The HOW of a DSPy program: the strategy used to ask the model. Predict asks directly, ChainOfThought reasons first, ReAct uses tools in a loop. All take the same signature.
optimizer (teleprompter)
Any DSPy component that improves a program using training examples and a metric. Historically called teleprompters, which is why the import path is dspy.teleprompt.
Predict
The simplest module: turn the signature into a prompt, ask once, parse the reply into output fields.
Prediction
The object a DSPy call returns. One attribute per output field: result.sentiment, result.urgency, and so on.
ReAct
Reason plus Act: a module where the model alternates between thinking and calling tools you provide, until it can answer. Named in the talk, not run live.
signature
The WHAT of a DSPy program: named, typed inputs and outputs, written as a string ("text -> sentiment: str") or a class. DSPy generates the actual prompt from it.
temperature
The randomness dial on generation. 0 means deterministic: the same input always yields the same output. Higher values add variety. Demos and evaluations want 0.
token / max_tokens
Models read and write in tokens, chunks of roughly three quarters of a word. max_tokens caps the response length, bounding cost and latency.
trainset
The labeled examples handed to the optimizer. Here, 12 examples for the live demo and 40 for the offline MIPROv2 compile.
Reference B

Glossary: Python bits used in the notebook

API key
A secret string that identifies your account to a model provider. Lives in .env, never in code, never on screen.
class
A blueprint for objects. The notebook uses classes two ways: to describe tasks (signatures) and once to wrap the hand-tuned prompt so it can be called like a function.
docstring
The triple-quoted string on the first line of a class or function. Normally documentation; in a DSPy signature it becomes the model's instruction.
.env and dotenv
A plain text file of KEY=value secrets next to your code, loaded at runtime by load_dotenv(). Gitignored, so secrets never reach a repository.
f-string
f"Hello {name}": put an f before the quotes and drop variables into braces. Format specs live after a colon, so {accuracy*100:.0f} means no decimal places.
import / from ... import
Bring a library (or one piece of it) into your program. import dspy loads the whole library; from dotenv import load_dotenv takes one tool out of the box.
keyword argument
Passing a value by name: classify(text="..."). DSPy leans on this so inputs match signature field names exactly.
list / list of objects
An ordered collection in square brackets. The train and test sets are lists of dspy.Example objects.
list comprehension
Build a list in one line by walking another: [name for name, _ in entries]. The underscore means "ignore this value."
raise RuntimeError
Stop the program on purpose with a readable message. Better a clear failure at setup than a confusing one later.
SimpleNamespace
A bare-bones object you can hang attributes on: SimpleNamespace(urgency="urgent").urgency. Used to make the hand-tuned wrapper's output look like a DSPy Prediction.
try / except
Attempt something that might fail; if the named error happens, run the fallback instead of crashing. The scoreboard uses it to survive a missing matplotlib.
type hint
The : str or : int after a name, labeling what kind of value it holds. In DSPy signatures, hints tell the parser what to convert model output into.
with statement
Creates a temporary context that cleans up after itself. with dspy.context(lm=big): swaps the model inside the block and restores the default automatically when the block ends.
__call__
A special method that lets an object be used like a function: defining it is why hand_tuned(transmission) works. Pronounced "dunder call."
